# 减少正式站重复“加载中”设计

## 背景与根因

正式站的静态资源由 CloudBase 返回 `Cache-Control: no-store`，浏览器刷新后会重新请求资源。应用同时把首页、广场和通知拆成懒加载模块，并让广场缓存只存在于当前 JavaScript 内存中。因此冷启动会显示首页加载提示，首次进入广场会等待模块和数据，刷新后广场内存缓存又会丢失；已登录用户还会在首页自动加载通知和 CloudBase SDK。

## 方案比较

1. **关键页面直接加载，通知按需加载，广场使用会话缓存（采用）**：消除常用路径的可见等待，同时保留后台、编辑器等重型低频页面的代码分包。
2. **全部页面直接打入主包**：交互时几乎没有模块加载提示，但会继续放大已经接近 500 KB 的主包，不适合当前无缓存的托管环境。
3. **只把文字换成骨架屏**：视觉上更柔和，但没有减少任何请求，也不能解决刷新后重复等待。

## 设计

### 首页

`HomeDiscoveryPage` 改为普通静态导入，并移除首页分支外层的 Suspense 加载提示。首页内容随主应用第一次渲染出现，不再单独请求首页分包，也不显示“正在打开首页…”。

### 通知

首页只渲染轻量铃铛按钮。只有用户点击铃铛时才动态导入通知面板，随后面板自身请求通知数据。取消通知组件挂载后的立即查询和 30 秒轮询，避免已登录用户仅访问首页就加载 CloudBase SDK。打开面板时仍显示面板内部的短暂加载状态，并保留手动重新打开后的刷新。

### 广场

广场列表改为普通静态导入，彻底移除第一次渲染时由 `React.lazy` 触发的“正在打开艺史广场…”提示。现有广场缓存增加带版本号的 `sessionStorage` 持久化：同一标签页刷新后先读缓存并立即渲染，再在后台请求最新数据。缓存按用户和主题隔离；解析失败或存储不可用时自动退回内存缓存。

### 部署边界

代码修改不能改变 CloudBase 托管层返回的 HTTP 缓存头。部署后仍需在 CloudBase 控制台为带哈希文件名的 `/assets/*` 设置长期缓存；`index.html` 保持短缓存或不缓存，以便新版本及时生效。

## 错误处理

- 会话缓存损坏时丢弃该条缓存，不影响广场正常请求。
- 后台刷新失败但存在缓存时继续展示缓存内容，并提示“内容刷新失败，当前显示上次结果”。
- 通知模块或请求失败时铃铛仍可用，不阻塞首页主体。

## 验收

- 自动测试验证首页不再懒加载、通知不再挂载即请求、会话缓存可跨模块重新载入读取且保持用户/主题隔离。
- 完整运行 `npm test` 和 `npm run build`。
- 本地浏览器验证首页无加载提示、进入广场后返回并刷新仍可立即显示缓存，控制台无相关错误。
