# 草稿删除与首图封面设计

## 目标

修复社区帖子编辑器的两个问题：

1. 用户可在“我的账户 → 我的草稿”中删除自己的草稿。
2. 用户选择图片后立即看到图片；排序后的第一张图片始终自动成为封面，不再需要手动设置封面。

## 当前根因

图片组件添加首批图片时，先调用图片列表回调，再调用封面回调。父组件的两个回调都基于同一个旧 `draft` 快照更新状态，后一次封面更新会把刚添加的图片列表覆盖回旧值，因此首批图片没有显示。

数据服务已经提供带用户归属校验的 `deletePost(userId, postId)`，但账户页草稿列表没有删除入口，也没有在删除成功后更新本地列表。

## 设计

### 图片与封面

- 将“图片列表”和“封面编号”视为一个原子变更，避免两个独立回调连续覆盖状态。
- 图片组件向编辑器返回完整结果：更新后的图片列表，以及由第一张图片推导出的 `coverImageId`。
- 添加图片后，第一张图片自动成为封面并立即显示。
- 拖动排序后，新的第一张图片自动成为封面。
- 删除图片后，剩余的第一张图片自动成为封面；没有图片时封面编号为空。
- 移除每张图片上的“设为封面”按钮；第一张图片显示“封面”标识。
- 发布前仍保留封面预览步骤，用于确认最终效果，不再要求用户选择封面。

### 删除草稿

- 在“我的账户 → 我的草稿”中，每条草稿保留点击主体进入编辑器的行为，并在右侧增加独立的“删除”按钮。
- 点击删除按钮后显示浏览器二次确认，避免误删。
- 确认后调用现有 `communityService.deletePost(userId, draftId)`。
- 删除成功后立即从账户页的草稿列表移除，不要求刷新页面。
- 删除失败或服务不可用时保留该草稿，并在草稿区域显示错误信息。
- 删除事件阻止冒泡，不能同时打开草稿编辑器。

## 数据流

图片选择、排序或删除：

1. 图片组件计算新的有序图片列表。
2. 图片组件以第一张图片的编号计算封面编号。
3. 编辑器通过一次状态更新同时写入 `images` 和 `coverImageId`。
4. 自动保存读取一致的草稿快照。

草稿删除：

1. 用户点击草稿右侧删除按钮并确认。
2. 账户页调用删除服务。
3. 服务校验当前用户拥有该草稿并将其标记为已删除。
4. 成功后账户页过滤对应草稿；失败则显示错误。

## 测试与验收

- 单元测试：新增图片变更结果以第一张图片为封面。
- 单元测试：重新排序后以新的第一张图片为封面。
- 单元测试：删除当前第一张图片后封面切换到剩余第一张；删除全部图片后封面为空。
- 服务测试：草稿所有者可以删除，其他用户不能删除。
- 页面结构测试：账户页提供草稿删除入口，并在成功后更新列表。
- 构建与现有测试全部通过。
- 浏览器验证：第一次选择图片立即显示；首图带封面标识；排序和删除后标识正确移动；删除草稿需确认且成功后条目立即消失。

## 范围边界

本次不增加草稿回收站、批量删除、自定义封面或撤销删除，也不改动已发布帖子的删除流程。
