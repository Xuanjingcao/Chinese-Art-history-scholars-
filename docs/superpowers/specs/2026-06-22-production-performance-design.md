# 正式站前端性能优化设计

## 背景

正式 CloudBase 站点的静态资源响应禁用了浏览器缓存，且首页当前会立即执行 CloudBase 健康检查。进入广场时，帖子与用户 reactions 又按顺序查询。部署平台缓存仍需后续单独配置，本阶段只处理仓库内可以确定改善的请求瀑布和重复请求。

## 目标

- 首页首屏不因状态徽标而下载 CloudBase SDK 或读取数据库。
- 登录用户进入广场时，帖子与 reactions 并行查询。
- 同一浏览器会话内再次进入相同话题时立即显示已有数据，并在后台刷新。
- 刷新失败时保留旧列表，不用空白页替换可用内容。

## 方案比较

### 方案 A：轻量模块缓存与后台刷新（采用）

使用项目现有 React 状态和模块级 Map，不引入依赖。缓存键由 `topic + userId` 组成，避免不同用户共享点赞状态。页面首次没有缓存时显示加载文案；命中缓存时立即渲染，并静默请求最新数据。

优点是改动小、可测试、不会增加正式包体；缺点是页面刷新后缓存会消失。

### 方案 B：引入 SWR 或 TanStack Query

可以获得缓存、去重和失效管理，但会增加依赖、包体和迁移范围。本阶段需求较小，不采用。

### 方案 C：Service Worker 缓存

可以绕过部分静态托管缓存问题，但生命周期和更新一致性复杂，而且不能减少首次 CloudBase 请求瀑布。本阶段不采用。

## 设计

### 首页 CloudBase 状态

`Header` 不再在挂载时调用 `getCloudBaseHealth()`。徽标仅根据运行配置显示“本地模式”或“云端后台”。真正需要数据库的功能仍沿用现有服务层错误处理，因此取消首屏探测不会隐藏功能请求本身的错误。

### 广场查询并行化

当存在 `userId` 时，`community_posts` 和该用户的 `community_reactions` 请求同时开始，通过 `Promise.all` 等待。帖子记录解析完成后再过滤与当前列表有关的 reactions。未登录用户只查询帖子。

### 会话缓存与后台刷新

新增一个聚焦的缓存模块，提供读取、写入和测试清理能力：

- 缓存键：话题和当前用户 ID。
- 缓存值：完整的 `CommunityPost[]`，包含当前用户点赞状态。
- 页面初始化命中缓存时直接显示列表且不出现全屏加载状态。
- 后台请求成功后替换页面数据并更新缓存。
- 后台请求失败且已有缓存时保留列表，并显示轻量错误提示；首次请求失败仍使用现有错误状态。
- 点赞的乐观更新和服务端回写同时同步缓存，避免离开再返回时状态倒退。

缓存只在当前页面进程有效，不写入 localStorage，避免跨版本旧数据和额外 schema 管理。

## 测试

- 单元测试缓存键隔离、读写副本和清理行为。
- 源码行为测试确认首页不再调用健康检查，CloudBase 帖子与 reactions 使用并行请求。
- 页面源码测试确认命中缓存时跳过首屏 loading、请求结果和点赞状态会写回缓存。
- 运行全量测试和生产构建。
- localhost 浏览器验证：首页无 CloudBase 首屏探测；广场首次加载后，返回再进入能立即显示帖子且无控制台错误。

## 非目标

- 不在本阶段修改 CloudBase 控制台、CDN 或静态资源响应头。
- 不增加离线能力或持久化缓存。
- 不更改帖子排序、权限、发布、评论或点赞语义。
