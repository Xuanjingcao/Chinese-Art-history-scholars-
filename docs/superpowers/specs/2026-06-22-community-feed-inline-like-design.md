# 广场卡片无刷新点赞设计

## 目标

登录用户可在“艺史广场”的帖子卡片上直接点赞或取消点赞，无需进入帖子详情，也不刷新或重新加载广场列表。

## 交互规则

- 点击卡片的标题、正文、封面等非操作区域，继续进入帖子详情。
- 点击红心只切换当前帖子的点赞状态，不触发卡片跳转。
- 未登录用户点赞时打开现有登录窗口，点赞数和红心状态不变。登录后由用户重新点击。
- 已登录用户点击后立即在本地更新红心和数字，CloudBase 写入在后台进行。
- 单条点赞请求未完成时，仅锁定该帖子的点赞按钮，不影响列表滚动和其他帖子。
- CloudBase 写入失败时回滚到操作前状态，并在卡片内显示“点赞失败，请重试”。

## 组件边界

`CommunityFeedPage` 继续作为广场帖子列表的状态拥有者：

- 获取帖子时传入当前用户 ID，使每条帖子包含 `likedByCurrentUser`。
- 实现点赞的乐观更新、请求中状态、失败回滚和错误提示。
- 只更新 `posts` 中命中的那一条记录，不重新请求广场列表。

`CommunityPostCard` 只负责展示和事件转发：

- 接收 `onLike`、`likePending` 和点赞错误状态。
- 卡片外层改为语义化容器，内部分离“打开详情”和“点赞”按钮，避免嵌套按钮。
- 点赞按钮提供明确的 `aria-label` 和 `aria-pressed`。

`App` 向广场页传入当前用户及现有登录回调，不新建第二套登录状态。

## 数据流

1. 用户点击某条卡片的红心。
2. 广场页检查登录状态和该帖子是否正在请求。
3. 已登录时保存旧值，立即反转 `likedByCurrentUser` 并增减 `likes`。
4. 调用 `communityService.toggleReaction(userId, postId, 'like')`。
5. 以服务返回的 `active` 作为最终点赞状态；如请求失败则恢复旧值。

## 测试与验收

- 卡片点赞按钮与打开详情交互相互独立。
- 未登录点赞只触发登录窗口。
- 已登录点赞和取消点赞均立即更新当前卡片，页面不刷新。
- 请求失败会回滚，且其他卡片仍可操作。
- 广场列表使用当前用户 ID 加载点赞状态。
- 在移动端和桌面端验证卡片跳转、红心状态、数字变化和控制台无新错误。
